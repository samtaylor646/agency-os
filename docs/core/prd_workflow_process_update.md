# Product Requirements Document (PRD): AgencyOS End-to-Task Workflow Process Update

## 1. Executive Summary
This PRD outlines the requirements for formalizing and enforcing the End-to-Task Workflow Process across the AgencyOS ecosystem. Based on the findings in `docs/operations/workflow_process_review.md`, this update introduces a strict five-phase agent lifecycle, culminating in a mandatory End of Task Sequence: **Docs/Memory -> HITL (Human-in-the-Loop) -> Git Master**. This update ensures absolute traceability, context preservation, quality assurance, and pristine version control.

## 2. Objective
To mitigate context loss, ensure rigorous quality control, and maintain a robust development pipeline by enforcing a standardized workflow protocol across all AgencyOS agents. The goal is to mathematically guarantee that no task is considered complete until documentation is updated, a human has formally approved via User Acceptance Testing (UAT), and the state is securely committed to version control.

## 3. Scope & Requirements

### 3.1 The Five-Phase Workflow
The workflow process will encompass the following five phases, which must be adhered to by all agents:

1. **Phase 1: Task Initiation & Routing:** `agents-orchestrator` must evaluate intent and route to the correct specialist using the `switch_mode` tool.
2. **Phase 2: Guardrails & Environment Initialization:** Specialists must validate against `settings.md` via `validation_layer.py` and isolate work in a new Git branch.
3. **Phase 3: Execution & Continuous Harvesting:** Implementation occurs with ad-hoc human spot-checks and continuous harvesting of repeatable workflows to the `agents/` directory.
4. **Phase 4: The QA Gate:** Automated tests are verified, and concise, step-by-step **Human Testing Instructions** are generated by the responsible agent or the `evidence-collector` for the upcoming UAT.
5. **Phase 5: The End of Task Mandate (Finalization):** This phase is triggered ONLY after passing the QA Gate.

### 3.2 Phase 5: End of Task Mandate Details
This is the core of the update, requiring strict sequential execution:
* **Step 1: Handoff Documentation & Memory Updates:** Agents MUST update `.roo/memory/changelog.md` and `.roo/memory/active_context.md`. New documentation must be strictly routed to the correct `docs/` subfolder (e.g., `docs/core`, `docs/technical`).
* **Step 2: HITL Verification (Formal UAT):** Agents MUST explicitly halt execution and present the human operator with the Human Testing Instructions from Phase 4. Explicit approval is required to proceed.
* **Step 3: Git Commit & Remote Push:** Following human approval, the agent MUST encapsulate all changes in a `git commit` on the isolated branch and push to the remote repository (`git push`).

## 4. Required Configuration Updates

To implement this workflow, the following configuration files must be updated:

### 4.1 Updates to `.clinerules`
* Add explicit routing rules mandating Phase 1 execution by the `agents-orchestrator`.
* Add strict branch isolation rules for Epic/Feature initiation.
* Mandate the generation of **Human Testing Instructions** prior to UAT.
* Implement the explicit Checklist Mandate for Phase 5 (Docs/Memory -> HITL -> Git Master), requiring agents to use the `update_todo_list` tool to enforce completion before allowing the use of the `attempt_completion` tool.

### 4.2 Updates to `config/settings.md`
* Update operational rules to codify the Five-Phase Workflow as the global standard.
* Define strict behavioral guidelines for agents to halt and demand explicit human input during Phase 5 (HITL).
* Emphasize the "No Unsolicited Advice" rule to prevent agents from bypassing the formal testing and approval gates.

### 4.3 Operational Guide Updates
* Create or update standard operating procedures in `docs/operations/` (e.g., a formal `git_workflow_sop.md`) to define the branch isolation and committing standards expected by the `git-workflow-master`.
* Establish templates for the **Human Testing Instructions** required in Phase 4.

## 5. Success Metrics
* 100% compliance with memory and documentation updates prior to task closure.
* Zero uncommitted or unpushed code remaining on local instances after task completion.
* Verifiable human approval logs for all merged features.
* Elimination of context loss between tasks/sprints due to the enforced memory update protocol.
