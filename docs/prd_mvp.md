# AgencyOS: MVP Product Requirements Document (PRD)

## 1. Objective
Define the core features, user stories, and acceptance criteria for the AgencyOS Minimum Viable Product (MVP). The goal of the MVP is to prove the core value proposition: allowing a user to manage a specialized AI workforce that can orchestrate, execute, and QA complex, multi-step tasks autonomously.

## 2. Target Audience
- **Founders / Solopreneurs:** Needing to execute cross-functional tasks (development, marketing, operations) without hiring a full human team.
- **Agency Owners:** Looking to automate standard delivery pipelines (e.g., website builds, SEO campaigns).
- **Developers / Tinkers:** Wanting an extensible, local-first multi-agent orchestration framework.

## 3. Scope of MVP

### 3.1 Core Features (In-Scope for MVP)
1. **The Discovery & Strategy Engine**
   - Interactive prompt system where Project Management agents ask clarifying questions based on a user's initial high-level goal.
   - Automatic generation of formal Strategy artifacts (PRD, Roadmap, Runbook) saved directly to the workspace before execution begins.

2. **Central Orchestrator (Backend Engine)**
   - Ability to parse the finalized Strategy artifacts and decompose them into sub-tasks.
   - Assignment of sub-tasks to specialized agents based on their defined roles in the `/agents` directory.
   - State management (tracking which tasks are pending, in-progress, in-review, or complete).

3. **Validation & Guardrail Layer**
   - Pre-execution rule validation (checking tasks against `settings.md` and `agent_base.yaml`).
   - "No Unsolicited Advice" enforcement.
   - Prevention of infinite loops during agent handoffs.

4. **Environment Scaffolding (Pre-Execution)**
   - Ability for DevOps agents to automatically generate initial repository files (e.g., `docker-compose.yml`, `.devcontainer`, `package.json`) based on the Strategy artifacts.

5. **Dev/QA Loop (Check & Balance)**
   - Hardcoded pairing of a "Maker" agent (e.g., Senior Developer) and a "Checker" agent (e.g., Evidence Collector / QA).
   - Tasks cannot be marked 'Complete' until the Checker agent approves the Maker agent's output.

6. **Command Center Dashboard (Frontend)**
   - A single-page application (React/Vite) showing active agents.
   - A chat/input interface for the Discovery Phase and human approval at Quality Gates.
   - A visual pipeline / Kanban board showing tasks moving through the system.

### 3.2 Out of Scope (Post-MVP)
- Visual Drag-and-Drop Pipeline Builder.
- Third-party integrations (GitHub, Slack, Jira API connections) - MVP will rely on local file system read/writes.
- Multi-tenant SaaS hosting (MVP is a local/Docker deployment).
- Self-optimizing agent prompts.

## 4. User Stories & Acceptance Criteria

### Epic 1: Task Initiation & Strategy Generation (The Discovery Phase)
**User Story:** As a User, I want the system to ask me clarifying questions about my high-level goal, so that it can generate comprehensive Strategy artifacts (PRD, Roadmap, Runbook) before any execution begins.
- **AC 1:** User enters an initial high-level prompt into the central prompt box.
- **AC 2:** The Orchestrator pauses and prompts the user with 2-5 clarifying questions generated by a Project Manager agent.
- **AC 3:** Upon user response, the system generates and saves a PRD, Roadmap, and either maps the goal to an existing runbook or creates a custom `.md` runbook in the workspace.
- **AC 4:** The UI updates to present these artifacts to the user for final approval (Quality Gate) before moving to Execution.

### Epic 2: The Dev/QA Loop Execution
**User Story:** As a User, I want agents to check each other's work so that I don't have to review every detail manually.
- **AC 1:** When an executing agent finishes a task, it automatically routes to a designated QA/Reviewer agent.
- **AC 2:** If the QA agent fails the work, it is routed back to the executing agent with feedback, without user intervention.
- **AC 3:** The UI visually indicates that a task is in the "Review" state and shows the feedback loop count.

### Epic 3: Guardrails and Stability
**User Story:** As a User, I want the system to enforce strict boundaries so that agents don't hallucinate or get stuck in loops.
- **AC 1:** System logs an error and halts (requesting human intervention) if a loop exceeds 3 iterations.
- **AC 2:** System rejects agent outputs that violate the rules defined in `settings.md`.

### Epic 4: Human-in-the-Loop (Quality Gates)
**User Story:** As a User, I want to give final approval before a major milestone is finalized so that I maintain control over the final output.
- **AC 1:** The pipeline pauses execution when reaching a predefined "Quality Gate".
- **AC 2:** The dashboard presents the current output to the user with "Approve" or "Reject/Modify" buttons.
- **AC 3:** If approved, the pipeline continues to the next phase.

## 5. Technical Constraints
- Backend must be Python-based using standard AI orchestration libraries (e.g., LangChain/AutoGen/CrewAI conceptually, implemented via `central_runner.py`).
- Frontend must be React (Vite) and styled with Tailwind CSS.
- Communication between frontend and backend via REST API or WebSockets (for real-time agent updates).
- Deployment via Docker Compose for easy local setup.
